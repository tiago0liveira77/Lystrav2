Plano de Implementação — Lystra

 Contexto

 Projeto Flutter/Dart SaaS de gestão de listas de compras partilhadas. O repositório está vazio (sem código Flutter ainda). Spec completa em CLAUDE.md. Stack decidida: Firebase
 (Auth + Firestore + FCM), get_it para DI, go_router para navegação, Material 3 com design system customizado. Plataformas: iOS + Android. Abordagem: MVP em 3 fases. Design visual
 obtido via DesignSync antes de qualquer implementação de UI.

 Arquitetura MVVM estrita definida em .agents/skills/flutter-apply-architecture-best-practices/SKILL.md — é a referência para todas as features.

 Fase 1 — MVP: Auth + Listas + Items + Shopping Mode

 Ordem de implementação dentro de cada feature: domain → data → ui (nunca ao contrário).

 1.1 — Domain Models (lib/domain/models/, todos com @freezed)

 - app_user.dart — uid, email, displayName, isPremium, householdId
 - category.dart — id, name, colorHex, iconCode
 - item.dart — id, name, categoryId, unit, ownerId
 - shopping_list.dart — id, name, ownerId, memberIds, createdAt, isArchived
 - list_entry.dart — id, listId, itemId, quantity, isChecked, addedAt, checkedAt

 Regra: domain/ não importa nada de data/ ou ui/.

 1.2 — Data Models (lib/data/models/, com @JsonSerializable)

 Espelham os domain models. Cada um tem fromFirestore(DocumentSnapshot), toFirestore() e toDomain(). IDs nunca armazenados dentro do documento Firestore — sempre injetados do
 doc.id.

 Ficheiros: app_user_model.dart, category_model.dart, item_model.dart, shopping_list_model.dart, list_entry_model.dart.

 1.3 — Services (lib/data/services/)

 - firebase_auth_service.dart — wrapper stateless sobre FirebaseAuth. Expõe Stream<User?> authStateChanges, signInWithEmail, registerWithEmail, signOut. Retorna tipos do Firebase
 SDK, não domain models.
 - firestore_service.dart — wrapper genérico sobre FirebaseFirestore. Paths pessoais: users/{uid}/items, users/{uid}/lists, etc. Expõe getDoc, setDoc, addDoc, updateDoc, deleteDoc,
 streamCollection.

 1.4 — Repositories (lib/data/repositories/)

 Recebem services via construtor. Transformam data models em domain models. Gerem cache em memória com invalidateCache().

 - auth_repository.dart — expõe Stream<AppUser?> authStateChanges, signIn, register, signOut
 - item_repository.dart — CRUD + cache de List<Item>; getter filteredItems(categoryId, query)
 - category_repository.dart — CRUD + cache de List<Category>
 - shopping_list_repository.dart — CRUD de ShoppingList
 - list_entry_repository.dart — CRUD de ListEntry por listId

 1.5 — Feature: Auth

 ├── view_models/auth_view_model.dart   # ChangeNotifier: isLoading, errorMessage, signIn(), register()
 └── views/login_view.dart

 Erros do FirebaseAuthException mapeados para mensagens PT em _mapAuthError(code). Go_router auth guard trata da navegação após login — o ViewModel não navega.

 Fase 3 — Premium + Household Sync

 3.1 — Domain Model: Household

 lib/domain/models/household.dart — id, name, ownerId, memberIds, members (List<HouseholdMember>).

 Path Firestore: /households/{householdId} (collection raiz, não sub-collection de user — necessário para acesso multi-utilizador).

 3.2 — Realtime Listeners (Streams)

 Repositories pessoais usam Future. Para listas partilhadas de household, adicionar Stream<List<ListEntry>> watchEntries(listId) via Firestore.snapshots().

 ShoppingViewModel na Fase 3 usa StreamSubscription em vez de Future. Cancelar em dispose().

 3.3 — Premium (RevenueCat)

 - lib/data/services/revenue_cat_service.dart — initialize(userId), purchasePremium(), checkPremiumStatus(). Chave pública RC não é segredo em mobile.
 - lib/data/repositories/premium_repository.dart — lê users/{uid}.isPremium do Firestore (source of truth server-side, escrita via Cloud Function).
 - Cloud Function onRevenueCatWebhook — recebe evento INITIAL_PURCHASE/RENEWAL, valida, escreve isPremium: true no Firestore. Nunca confiar no cliente para declarar premium.

 3.4 — FCM

 lib/data/services/fcm_service.dart — requestPermission, guardar FCM token em users/{uid}/fcmToken, handler de mensagens em foreground (SnackBar in-app). Inicializar após login
 confirmado.

 3.5 — Firestore Security Rules completas

 Funções helper: isAuthenticated(), isUser(uid), isPremium(uid) (lê Firestore), isHouseholdMember(householdId).

 Regras household: read para membros verificados; write apenas para owner premium. Sub-collections de household (lists/, entries/) acessíveis a todos os membros.

 3.6 — Feature: Profile/Settings

 lib/ui/features/profile/
 ├── view_models/profile_view_model.dart   # estado premium, toggle tema, upgrade, household CRUD
 └── views/profile_view.dart
 └── views/widgets/premium_banner.dart     # gradient card secondary/secondaryContainer, visível se !isPremium
 └── views/widgets/household_section.dart  # lista membros, botão convidar
 └── views/widgets/settings_section.dart

 Preferência de tema guardada em flutter_secure_storage, não SharedPreferences.

 Verificação F3: Dois utilizadores em household veem check-offs em tempo real (< 2s). Compra premium actualiza Firestore via webhook. FCM notification abre lista via deep link.
 Security Rules testadas com Firebase Emulator Suite (firebase emulators:start).

 ---
 Convenções Transversais

 - Erros: capturados nos ViewModels (String? errorMessage), nunca nas Views. Views exibem via SnackBar.
 - Validação de input: nos Repositories, antes de chamar Services. Nunca nas Views.
 - Dependência circular proibida: domain/ não importa data/ nem ui/. data/ não importa ui/.
 - Code generation: dart run build_runner build --delete-conflicting-outputs após qualquer @freezed, @JsonSerializable ou @GenerateMocks.
 - ViewModels: sempre registerFactory no get_it — nunca singleton.
 - Segurança: sem segredos no cliente. Chaves de backend apenas em Cloud Functions. Tokens sensíveis em flutter_secure_storage.
